# Luxury Housing Sales Analysis

Bengaluru Real Estate Analytics Pipeline

100,000+ Records | Python + MySQL + Power BI

## Problem Statement:

Build a complete real estate analytics solution using Python for advanced data cleaning, load the refined dataset into a SQL database, and use Power BI to directly connect to SQL and build a dashboard. The goal is to replicate a real-world enterprise-level data pipeline and analysis environment using a complex housing dataset with 1,00,000+ records.

## Tech Stack

| Tool | Purpose |
|------|---------|
| Python 3.10+ | Data cleaning, feature engineering, pipeline orchestration |
| Pandas | Data manipulation — filtering, grouping, imputation, derived columns |
| NumPy | Numerical operations — outlier capping, median calculation |
| MySQL | Data warehousing — stores cleaned 100K row dataset in structured schema |
| Power BI Desktop | Interactive dashboards — 10 visuals with slicers, drill-through, DAX |

## Setup Instructions

### Prerequisites

- Python 3.10+
- MySQL Server running locally
- Power BI Desktop (free from Microsoft)

### 1. Clone and Setup Virtual Environment

```bash
cd Luxury-Housing-Sales-Analysis
python -m venv .env

# Windows:
.env\Scripts\activate

# Linux/Mac:
source .env/bin/activate

pip install -r requirements.txt
```

### 2. Run Data Cleaning

```bash
python data_cleaning.py
```

This script performs the full cleaning and feature engineering pipeline:
- Loads raw CSV (100K+ rows) into a Pandas DataFrame
- Cleans Ticket_Price_Cr — removes currency symbols, caps outliers using IQR
- Normalizes text fields — title case for Micro_Market and Developer_Name, uppercase for Configuration
- Handles missing values — median imputation grouped by Micro_Market and Configuration
- Fixes invalid values — negative Unit_Size_Sqft values converted to NaN
- Derives Price_per_Sqft = (Ticket_Price_Cr * 10,000,000) / Unit_Size_Sqft
- Derives Quarter_Number (1–4) and Quarter_Label (e.g. 2024-Q2) from Purchase_Quarter date
- Derives Booking_Flag — binary 1/0 column indicating a confirmed booking
- Exports Luxury_Housing_Cleaned.csv ready for SQL loading

### 3. Load Data into MySQL

```bash
python load_to_sql.py
```

This script:
- Creates the luxury_housing_db database if it does not exist
- Loads Luxury_Housing_Cleaned.csv into the luxury_housing table via SQLAlchemy
- Prints validation results — row count, status distribution, top developers

Default MySQL credentials:
- Host: localhost
- User: root
- Password: pass@123

*To change credentials, edit load_to_sql.py in the main() function.*

### 4. Run SQL Validation Queries

```bash
mysql -u root -p luxury_housing_db < sql_scripts/validation_queries.sql
```

Or copy-paste individual queries from sql_scripts/validation_queries.sql into MySQL Workbench or CLI.

### 5. Power BI Dashboard

Follow powerbi_instructions.md to connect Power BI to MySQL, create all DAX measures, and build the 10 interactive dashboard visuals with slicers and drill-through.

## Data Pipeline Overview

The pipeline flows in 3 stages:

**Stage 1 — Raw CSV (100K+ rows)**
The raw Luxury_Housing_Bangalore.csv contains unclean data with currency symbols, mixed-case text, missing values, negative sizes, and no derived analytical columns.

**Stage 2 — data_cleaning.py**
Python script using Pandas and NumPy transforms the raw file into a clean, analysis-ready dataset. All feature engineering happens here — Price_per_Sqft, Quarter_Label, Quarter_Number, and Booking_Flag are all derived in this stage.

**Stage 3 — load_to_sql.py**
mysql.connector connects Python to the MySQL instance and bulk-inserts the cleaned DataFrame into luxury_housing_db.luxury_housing. The schema is defined in sql_scripts/schema.sql.

**Stage 4 — Power BI**
Power BI Desktop connects directly to MySQL. All DAX measures, calculated columns, and new tables for aggregation are defined inside Power BI — no further Python processing is needed at this stage.

## All DAX Measures — Complete Reference

The following measures are created in Power BI Data view. Each entry documents what the measure calculates, why it was created, and which visuals use it.

| Measure | Formula Summary | Why Created | Visuals Used |
|---------|-----------------|-------------|--------------|
| Booking Count | COUNTROWS FILTER Booking_Flag = 1 | Counts only confirmed bookings — the raw Booking_Flag column cannot be summed meaningfully in a visual context without this wrapper | Line chart (Q1), Donut chart (Q5), Stacked column (Q4, Q6), Clustered column (Q8), KPI cards (Q10) |
| Not Booked Count | COUNTROWS FILTER Booking_Flag = 0 | Needed to show the non-conversion side of the funnel alongside Booking Count — both together make the stacked bar sum to 100% | Stacked column (Q4), Stacked column (Q6), Clustered column (Q8) |
| Booking Conversion Rate | DIVIDE(Booked, Total) * 100 | Booking_Flag averages give a decimal; this measure produces a clean percentage for axis labels, KPI targets, and reference lines | Scatter plot (Q3), Stacked column (Q4, Q6), Map color saturation (Q9), KPI (Q10) |
| Not Booked Rate | DIVIDE(Not Booked Count, Total) * 100 | Mirror of Booking Conversion Rate — needed so both stacked segments always sum to exactly 100% in the 100% stacked column charts | Stacked column (Q4), Stacked column (Q6) |
| Total Ticket Sales | SUM(Ticket_Price_Cr) | Aggregates revenue across any filter context — used wherever total revenue by builder, quarter or market is needed | Bar chart (Q2), Matrix table (Q7), Bubble map size (Q9), KPI cards (Q10) |
| Avg Ticket Size | AVERAGE(Ticket_Price_Cr) | Average differs from total when comparing builders of different transaction volumes — needed for the dual-axis builder performance chart | Bar/line combo (Q2), Detail page cards (Q10) |
| Project Count | DISTINCTCOUNT(Project_Name) | Counts unique projects rather than transaction rows — used as bubble size in scatter plot so prolific projects appear larger | Scatter plot bubble size (Q3), Detail page (Q10) |
| Avg Amenity Score | AVERAGE(Amenity_Score) | Needed at project level for the scatter plot X-axis — row-level scores must be averaged per project before plotting | Scatter plot X-axis (Q3) |
| Revenue Target | AVERAGEX over Developer_Name | Sets a dynamic benchmark equal to the average builder revenue — KPI visuals need a target to show green/red directional indicators | KPI indicators (Q10) |
| Revenue Rank | RANKX ALL Developer_Name | Shows each builder's market rank (1st, 2nd, etc.) on the drill-through detail page — RANKX recalculates rank correctly under any filter context | Detail page table (Q10) |

## Dashboard Design Decisions — Why Each Choice Was Made

**Q3 — Amenity Impact: Why a New Project_Summary Table?**
The scatter plot requires one data point per project, not one per transaction row. With 100,000 rows, if you plot directly from the main table, Power BI draws a single dot representing the overall average of all 100,000 rows — which is meaningless for correlation analysis.
By creating Project_Summary via Power Query Group By, each row becomes one project with its pre-aggregated Avg_Amenity_Score, Booking_Rate, and Project_Count. This gives the scatter plot the correct granularity — one bubble per project — which is the only way to visually detect whether higher amenity scores correlate with better conversion rates across different projects.

**Q3 — Amenity Impact: Why Use Quarter_Label in the Scatter Plot?**
Quarter_Label is not used on the scatter plot axes — it is used in the slicer sitting alongside the chart. The purpose is to answer the time-dimension question: does the amenity-to-booking correlation hold consistently across all quarters, or did it strengthen or weaken over specific periods?
Without a quarter slicer, the scatter shows a blended view across all 8 quarters which may mask important seasonal patterns. A viewer can select 2024-Q1 alone and see if the correlation was stronger or weaker in that quarter versus 2023-Q4, making the analysis genuinely dynamic rather than a static snapshot.

**Q4 — Booking Conversion: Why Stack Booked and Not Booked on Top of Each Other?**
Stacking Booking Conversion Rate and Not Booked Rate on top of each other in a 100% stacked column chart serves one specific purpose — every column sums to exactly 100% regardless of how many transactions that micro-market has. This removes volume bias from the comparison.
If you used a simple clustered bar chart showing raw Booked Count, a micro-market with 6,400 transactions would always look bigger than one with 6,100 — even if both have identical 66.7% conversion rates. The 100% stack eliminates this distortion and makes the conversion rate comparison fair and accurate across all 15+ micro-markets.

**Q5 — Configuration Demand: Why a Text Box for Total Booked Count (66,659)?**
Power BI donut charts do not natively support centre-hole text labels. The donut hole is rendered as empty canvas space with no built-in field well for placing a value inside it. The only way to display a summary number (like total bookings = 66,659) visually centred inside the donut is to overlay a Text Box positioned manually over the hole.
This is a standard Power BI workaround. The Text Box is just a floating label — it does not update dynamically when slicers filter the chart. For a dynamic centre label that updates with filters, you would need a custom visual from AppSource. For this dashboard the static total is sufficient since the donut itself updates proportionally.

**Q5 — Configuration Demand: Why Add a Bar Chart Alongside the Donut?**
Donut charts are excellent for showing proportions but poor at communicating exact values. When three slices are nearly identical in size (33.5% vs 33.4% vs 33.1%), it is visually impossible to tell which is larger from the arc lengths alone — human perception of arc area is notoriously inaccurate.
The bar chart alongside shows the exact Booked Count numbers (22,340 vs 22,230 vs 22,089) as bar lengths, which are far easier to compare precisely. Together the two visuals complement each other — the donut communicates proportion at a glance, the bar chart confirms the exact ranking with readable numbers.

**Q5 — Configuration Demand: Why a Transaction Type Slicer?**
Transaction_Type (Primary vs Secondary) is a critical market segmentation dimension for real estate. Primary market means buying directly from the developer on new inventory. Secondary market means buying a resale property from a previous owner.
Configuration preferences differ significantly between these two buyer pools. Investors in the secondary market may prefer 3BHK for liquidity, while end-users buying primary may prefer 4BHK and 5BHK+ for family use. Without the Transaction_Type slicer, the configuration donut shows a blended view that obscures these distinct demand patterns. The slicer lets the analyst separate and compare the two markets independently.

**Q7 — Quarterly Builder Contribution: Why Use Drill Down in the Matrix?**
The matrix table at its top level shows Developer_Name as rows and Quarter_Label as columns — one revenue figure per builder per quarter. Drill down adds Micro_Market as a second level beneath each builder row.
This enables a critical analysis question: is a builder's quarterly revenue concentrated in one micro-market, or is it spread evenly across Bengaluru? For example, Prestige might dominate Sarjapur Road in 2024-Q1 but be weak in Whitefield. Without drill down, this geographic concentration within a builder's quarterly performance is completely invisible. The drill down converts a flat revenue table into a two-level hierarchy that reveals both builder performance and geographic strategy simultaneously.

**Q8 — Possession Status: Why Two Slicers?**
The chart answers how possession status affects both buyer type and booking decisions simultaneously — two distinct analytical dimensions. A single slicer can only filter one dimension at a time.
The Possession_Status slicer (tile style, 3 values) isolates one construction stage at a time — showing the full buyer type breakdown for just Ready to Move properties. The Buyer_Type slicer (dropdown, multi-select) then lets you go deeper — selecting NRI and HNI together to compare whether these two high-value segments behave differently across possession stages. Together they enable two-dimensional filtering that a single slicer cannot provide.

**Q8 — Possession Status: Why Three Separate Visuals?**
Each of the three visuals in Q8 answers a different dimension of the same question:
Visual 1 — Clustered column chart (Booked Count by Possession Status, colored by Buyer_Type): Shows the raw volume of bookings. Answers which possession stage has the most absolute demand and which buyer type dominates each stage.
Visual 2 — Clustered column chart (Not Booked Count): Shows the non-conversion side. Without this, you cannot see whether a possession stage has high volume because buyers are genuinely buying, or just browsing. Comparing Visual 1 and Visual 2 side by side reveals the full funnel.
Visual 3 — Line and Clustered Column combo (Booked Count as bars, Booking Conversion Rate as line): Combines volume and efficiency on one axis. A possession stage can have high booking count but poor conversion rate (large market, low commitment) or low count but excellent conversion (niche market, highly decisive buyers). This dual-axis view is the only way to see both dimensions simultaneously.

**Q9 — Geographical Insights: Why Add a New Full_Location Column?**
Power BI's built-in geocoding looks up location names globally, not just within India. The name Whitefield exists in multiple countries and cities worldwide. Without context, Power BI may geocode it to a location in the United States or elsewhere, placing your bubble in completely the wrong position on the world map.
Adding a calculated column that appends ', Bangalore, India' to every micro-market name (e.g. 'Whitefield, Bangalore, India') gives the geocoder enough context to always resolve to the correct Bengaluru suburb. This single column addition is what makes the map geographically accurate. The manual lat/long SWITCH formula is added as a fallback for cases where even the full name geocodes incorrectly due to Bing Maps data gaps for Indian suburbs.

**Q10 — Top Performers: Why Create a Separate Top 5 Builders Table?**
The main output table has 100,000 rows across 15+ builders. If you apply a Top N visual-level filter directly on a Card visual, the filter only suppresses display but does not change what the measure calculates — the card still computes against all 100,000 rows unless the filter context is properly isolated.
Creating a Top5_Builders calculated table using TOPN and SUMMARIZE physically materialises a 5-row table in the data model with pre-computed revenue, booked count, and booking rate for exactly those 5 builders. This gives stable, correctly isolated measure calculations for the KPI cards, ensures the drill-through detail page receives the right filter context, and makes the Revenue Rank measure using RANKX compute correctly within the top-5 context rather than ranking against all 15+ builders.
